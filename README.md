# Java-TS-Gen [JTG]

**Annotation-driven TypeScript generation for Java records.**

JTG is a developer-time Maven plugin. You annotate a Java `record` with `@TsRecord`, run the goal, and a `.ts` file is written next to the Java source. With an execution bound to the lifecycle, generation runs on every build; otherwise invoke it explicitly.

```bash
mvn jtg:generate
```

Published version (see the root `pom.xml` for the current release): **0.1.1**.

---

## Quick Start

### 1. Add JTG to your project

In your `pom.xml`:

```xml
<dependencies>
  <!-- The annotation (SOURCE retention â€” zero runtime overhead) -->
  <dependency>
    <groupId>io.github.loomforge</groupId>
    <artifactId>jtg-annotation</artifactId>
    <version>${version}</version>
    <scope>provided</scope>
    <optional>true</optional>
  </dependency>
</dependencies>

<build>
  <plugins>
    <plugin>
      <groupId>io.github.loomforge</groupId>
      <artifactId>jtg-maven-plugin</artifactId>
      <version>${version}</version>
      <executions>
        <execution>
          <phase>generate-sources</phase>
          <goals>
            <goal>generate</goal>
          </goals>
        </execution>
      </executions>
    </plugin>
  </plugins>
</build>
```

You may omit `<phase>`: the goalâ€™s default phase is `generate-sources`.

If you declare the plugin **without** `<executions>`, nothing runs until you execute `mvn jtg:generate` yourself.

### 2. Annotate your records

```java
import io.github.loomforge.jtg.annotation.TsRecord;

@TsRecord
public record User(String name, int age, boolean active, UUID id) {}
```

### 3. Run

With the execution above, a normal build runs generation before compilation:

```bash
mvn compile       # jtg:generate runs on generate-sources
mvn package       # same
mvn jtg:generate  # explicit invocation anytime
```

**Result** â€” `User.ts` is created in the same directory as `User.java`:

```typescript
// Generated by JTG on 2026-04-08 12:00:00
// Source: User.java
// Do not edit this file manually - changes will be overwritten.

export interface User {
  name: string;
  age: number;
  active: boolean;
  id: string;
}
```

## Same-file record dependencies

If a `@TsRecord` type references **another record declared in the same `.java` file** (by simple name, including inside generics such as `List<Address>`), JTG emits those dependency types **first in the same `.ts` file**, then the annotated record. Types from **other files** are not followed yet; they are emitted as written in source (see type mappings).

## Annotation Options

```java
@TsRecord(
    exportName = "UserDTO",   // TypeScript name (default: Java record name)
    asType     = true         // Emit `type` alias instead of `interface`
)
public record User(String name, int age) {}
```

**With `asType = true`:**

```typescript
export type UserDTO = {
  name: string;
  age: number;
};
```

## Type Mappings

Mappings are based on the **simple** type name as it appears in source (qualified names such as `java.util.UUID` are normalized to `UUID`).

| Java                                                                       | TypeScript                                                     |
|----------------------------------------------------------------------------|----------------------------------------------------------------|
| `byte`, `short`, `int`, `long`, `float`, `double`                          | `number`                                                       |
| `Integer`, `Long`, `Float`, `Double`, `BigDecimal`, `BigInteger`           | `number`                                                       |
| `boolean`, `Boolean`                                                       | `boolean`                                                      |
| `char`, `Character`, `String`                                              | `string`                                                       |
| `UUID`                                                                     | `string`                                                       |
| `LocalDate`, `LocalDateTime`, `ZonedDateTime`, `OffsetDateTime`, `Instant` | `string`                                                       |
| `Object`                                                                   | `unknown`                                                      |
| `List<T>`, `Set<T>`, `Collection<T>`                                       | `T[]` (after mapping `T`)                                      |
| `Map<K, V>`                                                                | `Record<string, V>` (value type mapped; key treated as string) |
| `Optional` / `Optional<T>`                                                 | `unknown \| null` (inner `T` is not unfolded yet)              |
| Unknown / other types                                                      | emitted as the simple name (e.g. `MyDto`)                      |

Java array types `T[]` become TypeScript `T[]` after mapping the element type.

## Plugin Configuration

Optional parameters:

```xml
<plugin>
  <groupId>io.github.loomforge</groupId>
  <artifactId>jtg-maven-plugin</artifactId>
  <version>${version}</version>
  <configuration>
    <!-- Extra roots to scan (in addition to compile source roots) -->
    <additionalSourceDirs>
      <dir>src/api/java</dir>
    </additionalSourceDirs>

    <!-- Log every scanned .java path (at DEBUG) -->
    <verbose>true</verbose>

    <!-- Skip all generation -->
    <skip>false</skip>
  </configuration>
</plugin>
```

Override the phase if needed (for example bind to `process-sources`):

```xml
<executions>
  <execution>
    <id>jtg-generate</id>
    <phase>process-sources</phase>
    <goals><goal>generate</goal></goals>
  </execution>
</executions>
```

Command-line overrides:

```bash
mvn compile -Djtg.skip=true      # skip generation
mvn compile -Djtg.verbose=true   # verbose scanning
```

Property names: `jtg.skip`, `jtg.verbose`, `jtg.additionalSourceDirs` (Maven maps nested config to the property where supported).
